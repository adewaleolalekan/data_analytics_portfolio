Right Tool for the Job

A case study on selecting the appropriate data analysis tool — Spreadsheets vs. SQL — based on real-world business scenarios.

4 real-world scenarios
SQL vs. Spreadsheet decisions
Clear justifications for each

Overview

One of the most important decisions a data analyst makes is choosing the right tool before starting any analysis. Using SQL on a 300-row CSV wastes time; using a spreadsheet on a multi-table relational database is impractical. This case study walks through four real business scenarios and explains the reasoning behind each tool choice.

Scenarios

01

Identifying Top 10 Pet Supply Products

A major online pet supplies business needs to identify the top 10 products with the highest sales in the last month. Data is spread across multiple tables in the company's database — one for product details, another for sales data.

SQL

The data lives in a relational database and the task requires joining multiple tables. SQL is built for exactly this — it handles complex joins and aggregations across large datasets efficiently, far better than any spreadsheet could.

02

Determining a New Store Location

A small fitness tech company needs to identify which cities have the largest customer population to decide on a new retail location. The data is a single CSV file with 300 customer addresses. A simple bar chart is also required.

Spreadsheet

The dataset is small (300 rows) and comes in a simple CSV. A spreadsheet makes it easy to sort, filter by city, count customers, and generate a bar chart — all without needing a database setup. SQL would be overkill here.

03

Analysing Customer Satisfaction Surveys

A fashion retailer's marketing team has customer satisfaction survey results in a CSV file. They need to calculate average satisfaction scores and segment responses by demographics (age, gender, location) to tailor marketing strategies.

Spreadsheet

For a typical survey CSV, a spreadsheet's PivotTables make it easy to calculate average scores and segment by demographics without complex setup. SQL could work if the dataset were massive, but is unnecessary here.

04

Calculating Course Completion Rates

An online education platform needs to identify courses with the lowest completion rates. Data is stored in a relational database with separate tables for user registration, course enrollment, and course completion.

SQL

A classic SQL use case. Calculating completion rates requires joining across three tables (registration, enrollment, completion). SQL handles these joins and aggregations efficiently at scale — a spreadsheet simply cannot match this.

Decision Framework Summary

🗄️ Choose SQL when…

  • Data lives in a relational database
  • You need to join multiple tables
  • The dataset is very large
  • You need repeatable, automated queries

📊 Choose a Spreadsheet when…

  • Data comes as a simple CSV or flat file
  • The dataset is small (a few thousand rows or fewer)
  • You need quick charts or pivot tables
  • No database setup is available or needed
